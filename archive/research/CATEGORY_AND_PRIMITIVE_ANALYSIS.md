## 0. The honest finding, stated up front

The project is **not** discovering a new primitive in the *item* sense. The candidate primitives decompose as follows:

- The *item* primitive (attenuable, offline-verifiable delegation capability token) **already exists** as macaroons (Google, 2014) and the actively-maintained Biscuit token format — admitted in the project's own frozen `PRODUCT_THESIS.md` §P5 §3, verbatim: *"The attenuable, offline-verifiable delegation capability already exists as a primitive."*
- The candidate novelty is a *composition* of three existing primitives (SPIFFE workload identity + macaroon/biscuit-style attenuable capability + RFC 8693 token exchange / OAuth Status List revocation), **standardized** into a single portable, cross-trust-domain unit, **adopted** by parties other than its creator. That is precisely the *Docker* pattern — and is precisely the *compositional* novelty class, which the directive's taxonomy explicitly distinguishes from scientific/engineering/protocol/infrastructure novelty.
- The **one** load-bearing engineering piece that is genuinely unsolved by standardized composition — offline, partition-tolerant, *independent* revocation of a delegation token with no live call to the issuer and no third-party decision-making broker (C4 in the team's frozen `LEVEL0_1_FEASIBILITY_GATE.md`) — is, under the strict reading of the falsification hypothesis, an **information-theoretic limit, not a technology gap**: no technology, present or future, can let a relying party observe a revocation performed on the other side of a partition that isolated it from the issuer at revocation time, within a window shorter than partition-recovery. Under a relaxed reading, C4 collapses to "a thin revocation layer over OAuth Status List," which is composition over an existing IETF draft, not a new primitive.
- Realized gravity today is **A (none)** — `ECOSYSTEM_THESIS.md` §P5 §10, verbatim: *"Today, honestly: nothing would notice, because it doesn't exist yet."* The "if built and adopted" gravity is **B with a structurally credible path to C** — which is exactly what the team's own frozen document concluded.

**This document is the falsification verdict.** The framing of the work as "discovering a new primitive" is falsified at the *item* level (the item exists) and reduced to a *compositional + adoption-contingent* claim. The recommendation in §6.5 follows from that falsification: **Narrow** the framing to what the evidence actually supports, resolve the named scope ambiguities before any further architecture work, and instrument the one failure mode the framework is currently blind to (adoption-indifference), without continuing as if the primitive claim were established and without pivoting to a different problem.

## 1. Candidate primitives

Four candidate primitives were enumerated against the product definition and RFC-002's conceptual model. Each is treated with the falsification discipline the directive requires.

### P-A. Attenuable, offline-verifiable delegation capability token

- **Precise definition.** A cryptographic token attesting "X acting on behalf of Y, scope S, valid until T," that is (a) *attenuable* (the holder can narrow the scope, bound the TTL, or add caveats without contact with the issuer), (b) *chained* (capability can be passed onward with progressively-narrowed rights), and (c) *offline-verifiable* (verifier needs only the token and the issuer's public trust material; no live call to a broker).
- **Already exists?** **Yes.** Google's macaroons (2014, paper + deployed) and the actively-maintained Biscuit token format both implement this. RFC 8693 token exchange provides the standardized `act` claim for delegation format and supports nested `act` for chains. The project's own frozen `PRODUCT_THESIS.md` §P5 §3 states this verbatim: *"The attenuable, offline-verifiable delegation capability already exists as a primitive."*
- **Existing equivalents / adjacent concepts.** Macaroons (Google 2014), Biscuit tokens, RFC 8693 `act` claim (and nested `act`), SD-JWT (selective-disclosure JWTs), DPoP (demonstrating possession of an access token), OAuth access tokens with scope claims, X.509 proxy certificates (RFC 3820), Kerberos constrained-delegation tickets, AWS STS session policies.
- **Why those are insufficient.** They are not insufficient *as the item* — they ARE the item. They are insufficient *as the cross-trust-domain + workload-identity-bound + no-shared-broker + standardized composition* — that is, as the *frame*, not the *token*. That insufficiency is the candidate of P-B, not P-A.
- **New capability if P-A exists.** None additional — it exists.
- **What disappears if P-A never exists.** Nothing — it exists.
- **Novelty class.** **Not a candidate for novelty.** Prior art, substrate to build on.

**Falsification verdict for P-A.** The *item* is not novel. Any framing that treats P-A as the novel contribution is contradicted by the project's own frozen `PRODUCT_THESIS.md`. The novelty must be searched for elsewhere (P-B).

### P-B. Standardized binding of SPIFFE workload identity × attenuable delegation capability, portable across independently-operated trust domains with no shared authority

- **Precise definition.** A single, standardized, *interoperable* object that combines (a) a SPIFFE workload-identity credential (SVID) for the delegate X, (b) an attenuable, offline-verifiable capability (P-A) carrying the principal Y's authorization and the scope S / TTL T, and (c) portability rules that let a relying party in trust domain B verify a delegation issued by trust domain A, with no shared authority, no live broker call, and no mandatory new protocol adoption from third parties. This is the candidate the project's own frozen `PRODUCT_THESIS.md` §P5 §3 names, verbatim: *"a standardized binding between workload identity (SPIFFE) and attenuable capability tokens, portable across independently-operated trust domains with no shared authority."*
- **Already exists?** **No.** The components all exist independently; the *binding* and its *standardization across trust domains with no shared authority* do not. Multiple independent 2026 sources (cited in frozen `TECHNICAL_VALIDATION.md` §P5 §3, §P5 §6) converge on this gap: a named academic paper (Agent Identity Protocol), four IETF agent-identity drafts in early 2026, and a proprietary partial answer from Teleport (March 2026) are all working this edge because the open, interoperable standard does not yet exist.
- **Existing equivalents / adjacent concepts.** SPIFFE/SPIRE (graduated CNCF, authentication only, **deliberately scoped to exclude authorization** per `ECOSYSTEM_THESIS.md` §P5 §2); RFC 8693 token exchange (the `act` format is solved; nested prior-actors are *informational only* — not per-hop authorization gates, per fetched `LEVEL0_1_FEASIBILITY_GATE.md` S5/C11); macaroon/biscuit (the capability primitive, not bound to workload identity, not standardized for cross-org use); Kerberos constrained delegation (single-realm-anchored, online KDC); SPIFFE Federation (cross-trust-domain trust-bundle exchange, *explicitly disabled in the falsification experiment* because it smuggles a shared broker — see `P5_FALSIFICATION_EXPERIMENT.md` §3.2); AWS IAM Roles Anywhere (single-cloud-anchored); HashiCorp Vault dynamic secrets (broker-mediated).
- **Why those are insufficient.** Each covers an axis but not the intersection. SPIFFE has identity, not authorization/delegation. RFC 8693 has the delegation format, not per-hop authorization gates. Macaroon/biscuit has the capability, not workload identity. Kerberos and SPIFFE Federation have cross-domain, but are anchored to a shared authority (KDC / federation bundle) that the project explicitly forbids. The intersection — workload identity + attenuable capability + cross-trust-domain + no-shared-authority + standardized + no-new-protocol — is not claimed by any single listed standard.
- **New capability if P-B exists.** Genuinely least-privilege AI agent systems carrying a narrowly-scoped, time-bound, independently-revocable capability instead of a blanket credential. Cross-organizational workflows that don't require a shared IdP or federation setup. Structural cryptographic audit trails instead of log-reconstructed ones. Failure containment that revokes one delegated capability without restarting a workload.
- **What disappears if P-B never exists.** Agents regress to blanket credentials (long-lived API keys, human-equivalent tokens) or to operationally-expensive SPIFFE Federation (which the team's own `TECHNICAL_VALIDATION.md` §P5 §3 cites as "everyone wants SPIFFE, almost no one can afford to build it right"). Cross-organizational workflows require federation setup per pair. Audit stays log-reconstructed (a SO8/AP8 gap named in `RFC-002-conceptual-domain-model.md`).
- **Novelty class.** **Compositional.** This is the most important single finding in this document: P-B is *not* scientific / engineering / protocol / infrastructure / developer-experience / operational novelty. It is **compositional** — a new *load-bearing combination* of existing primitives, standardized into a unit, that would function as a new substrate **if** adopted. The directive's taxonomy explicitly lists "compositional" as a separate class. Docker was compositional novelty; P-B is of the same class.

**Falsification verdict for P-B.** P-B is the only candidate that is *not falsified* as a novelty — but its novelty is *compositional*, contingent entirely on third-party adoption, with zero current adoption signal. The framing "we are discovering a new primitive" is honest only if "primitive" is read in the *compositional* sense (the Docker sense) and only with the adoption contingency front-and-center. Reading "primitive" in any of the other six classes mis-states the claim.

### P-C. Offline, partition-tolerant, *independent* revocation of a delegation token with no live call to the issuer and no third-party decision-making broker (C4 in the frozen feasibility gate)

- **Precise definition.** A mechanism by which a relying party in trust domain B, potentially partitioned from the issuing party in domain A and from any third-party broker, can **observe, in-window** (within R seconds, R < delegation TTL), that a specific delegation token issued earlier has been **revoked** — where the revocation was performed on the issuer side after the token was issued, and the RP's observation must use only signed artifacts the RP already holds or can fetch over a passive signed-blob cache, and must not require a live call to any service in domain A nor a decision by a third party.
- **Already exists?** **No, in the strict reading — and the strict reading is a logical impossibility.** The frozen `LEVEL0_1_FEASIBILITY_GATE.md` (2026-07-04) falsified every candidate primitive against H₁'s strict no-live-call / no-broker clauses on the basis of fetched primary sources: RFC 7009 (exclusively online endpoint, fetched); draft-ietf-oauth-status-list (eventual consistency on the order of the refresh interval, refresh fetch is a live call to a service in domain A or a broker, fetched); short-TTL (expiry, not revocation). The gate also identified **S4 — a logical impossibility, not a technology gap**: *"no technology — present or future — can let an RP observe a revocation performed on the other side of a partition that isolated the RP from the issuer at the time of revocation, within a window shorter than partition-recovery. Strictly read, T4 ∩ T5 is a logical impossibility, not a technology gap."* (S4, verbatim.)
- **Existing equivalents / adjacent concepts.** RFC 7009 (online-only endpoint); OAuth Status List (eventual-consistency, periodic fetch from issuer URI, fetched); CRL/OCSP (online, or stale, or partitioned); short-TTL (expiry, not revocation); push-revocation (no standard protocol, no interop with SPIFFE/RFC 8693); cryptographic accumulators (Camenisch-Lysyanskaya-style, academic, no standardized OSS); macaroon attenuation (narrowing, not nullification).
- **Why those are insufficient.** Either (a) they require a live call (forbidden by H₁), (b) they deliver eventual consistency at a refresh interval (forbidden by T4's 2-second in-window requirement), (c) they are unstandardized, or (d) they deliver expiry or narrowing, not revocation. The falsification experiment's substrate (frozen `P5_FALSIFICATION_EXPERIMENT.md` §3.2) explicitly locks federation out and pins delegation TTL = 30 s, so short-TTL is also out by experimental design.
- **New capability if P-C exists.** Genuine in-window independent revocation during partition — which, under the strict reading, **cannot exist by information-theoretic limit**. Under a relaxed reading (revocation observable post-partition-recovery, or in-window only when partition is absent), C4 collapses to "a thin revocation layer over OAuth Status List" — which is composition over an existing IETF draft, not a new primitive.
- **What disappears if P-C never exists.** The strict-read claim disappears, which is what the project's own feasibility gate already documented. The relaxed-read claim survives as a configuration of OAuth Status List + a thin layer. No new capability is *uniquely* lost; existing drafts plus current primitives cover the relaxed case.
- **Novelty class.** **Either (a) an impossibility (strict reading) or (b) protocol / compositional (relaxed reading, but collapses to a configuration of an existing IETF draft).** Not a clean novel primitive in either reading. This is the **single most important falsification finding in this document** — the one candidate that the team's own evidence (the feasibility gate) has already, in effect, falsified.

**Falsification verdict for P-C.** P-C is not a novel primitive. It is either an information-theoretic limit (strict) or a configuration of OAuth Status List (relaxed). Any framing that treats P-C as a new primitive is contradicted by the team's own frozen `LEVEL0_1_FEASIBILITY_GATE.md`, written one day before this document.

### P-D. Reconstruction Record (tamper-evident, self-sufficient, offline third-party-verifiable record of a delegated act)

- **Precise definition.** A record of a delegation event that is (a) tamper-evident (any alteration is detectable by a third party), (b) self-sufficient (carries everything needed for offline verification), and (c) third-party-verifiable (an independent auditor can verify it without trusting the issuer or the relying party). This is the β spike-outcome primitive in the unfrozen `RFC-002-conceptual-domain-model.md` and supports the SO8 / AP8 audit concern the conceptual model names.
- **Already exists?** **Partially, as composing pieces.** Tamper-evident signed audit logs (e.g., `journald` + signed checkpoints); W3C Verifiable Credentials proof chains; Sigstore Rekor (signed transparency log of supply-chain artifacts); Certificate Transparency logs; signed JWT/JWS chains (RFC 7515/7519); Biscuit's offline-verifiable token structure. The *composition* of these into a delegation-specific audit record with third-party-verifiable revocation is not, to the project's evidence base, a standardized primitive.
- **Existing equivalents / adjacent concepts.** Sigstore/Rekor (artifact provenance, not delegation); Certificate Transparency (X.509 issuance, not delegation); W3C Verifiable Credentials (proof chains, not bound to delegation-with-independent-revocation); RFC 8693 nested `act` as a *provenance trail* (the `LEVEL0_1_FEASIBILITY_GATE.md` S5 confirms this is solved as a format, but only as provenance, not as per-hop authorization).
- **Why those are insufficient.** Each covers a piece but not the composed object. VC proof chains are closest but not bound to *delegation with independent revocation* — VC revocation is a different model (status list, similar to OAuth Status List, with the same partition limits named in P-C). Sigstore is about *artifact* provenance, not *delegation* provenance. RFC 8693 nested `act` is a *provenance trail* the verifier must trust, not a third-party-auditable revocation record.
- **New capability if P-D exists.** A third-party-auditable, structural cryptographic record of a delegation, not a log-reconstructed one. This is the SO8 / AP8 gap that `RFC-002-conceptual-domain-model.md` names. It would support post-hoc review of a delegation without trusting either the issuer or the relying party.
- **What disappears if P-D never exists.** Audit trails remain log-reconstructed (the same gap as the broader P-B "what disappears" — audit is one axis of P-B's promised benefits). No new capability is *uniquely* lost that the listed existing systems do not already cover, *if* the verifier is willing to trust the existing primitives' audit mechanisms (signed logs, CT-style transparency).
- **Novelty class.** **Compositional, derivative of P-B.** P-D is the audit view of the same composition. It does not introduce an independent novelty; it introduces an audit-angle on P-B.

**Falsification verdict for P-D.** P-D is not an independent novelty. It is the audit view of the P-B composition, and it inherits P-B's compositional novelty class plus P-C's revocation limit. It does not falsify the broader finding.

| 11 | **SPIFFE Federation** | Cross-trust-domain trust-bundle exchange | Solves *trust-bundle* exchange, not *delegation without a shared broker*; operationally expensive + incomplete (`TECHNICAL_VALIDATION.md` §P5 §3); *explicitly disabled* in the falsification experiment because it smuggles a shared broker (`P5_FALSIFICATION_EXPERIMENT.md` §3.2, verbatim: *"federation is the closest SPIRE-shipped mechanism that would otherwise smuggle a shared broker back in; we lock it out"*). | **Strongest adjacent — and explicitly excluded by experimental design.** Genuinely new relative to Federation? Yes — P-B's no-broker clause is *narrower* than Federation's design; the project is occupying a slot Federation cannot. |
| 12 | **Google Zanzibar** | Relationship-based authorization (Google-internal, paper + open-source derivatives); centralized, online, consensus-replicated tuple store | Centralized, *online*; architecturally the *opposite* pole from P-B's offline-self-sufficient-token model. No offline, no attenuable token, no delegation chain as a portable object. | **Adjacent conceptually (authorization), architecturally opposite (online centralized vs offline decentralized).** Genuinely new? P-B is the offline-decentralized answer Zanzibar explicitly is not. |

**Standards verdict.** The project is not re-inventing anything any single listed standard does. It is genuinely occupying an *intersection* (workload identity + attenuable capability + cross-trust-domain + no-shared-authority + no-mandatory-new-protocol + independently-revocable + offline-verifiable) that no listed standard claims. **But** — the unsolved status of that intersection is *partly* because one corner of it is a **logical impossibility** (P-C, in-window revocation during partition), not because anyone missed it. The "novel intersection" has a no-go corner. The "newness" is the *cross-section*, not any component within it; and the cross-section is compositionally novel but not *scientifically* or *engineeringly* novel.

**A second pattern the table makes visible.** Of the 12 systems, only Vault, AWS IAM Roles Anywhere, and SPIFFE Federation are "adjacent but genuinely insufficient" rather than "out of scope" or "orthogonal" — and all three are **broker-mediated, single-anchor, or single-cloud**. The cluster of insufficient-adjacent systems is *all broker-anchored*. The project's distinctive move is exactly the "no shared authority at the moment of verification" clause — that clause is what rules out the insufficient-adjacent cluster, and is also the clause that *solves* the problem those adjacent systems *can't* solve. The clause is the load-bearing novelty. The novelty is the *constraint*, not a new capability — which is a peculiar kind of novelty and further evidence that the "primitive" claim is best read as *compositional* (a constraint-defined composition) rather than as a new technology.

## 4. Sequential five-lens review

Lens-doc paths: `agents/agents/council/{empiricist,cartographer,red-team,economist,operator}.md`. Each lens is applied in its own voice, in the project's required sequence, with confidence labels and change-conditions per Empiricist doctrine.

### 4.1 Empiricist (Seat of Evidence) — opens

**Mental models used:** baseline test; counterfactual; placebo discipline; survivorship-bias audit; base rate.

- **Baseline test.** The baseline is: the *item* primitive (P-A, attenuable offline-verifiable delegation token) **already exists** (macaroons 2014, Biscuit). The project's own frozen `PRODUCT_THESIS.md` §P5 §3 states this verbatim. The static credential is the existing industry primitive underneath SPIFFE/SPIRE itself. SPIFFE is graduated CNCF and deployed at named scale (Uber, Stripe, Netflix per `TECHNICAL_VALIDATION.md` §P5 §3 Proven). The composable hypothesis is the baseline, not the alternative. *"What is the baseline?"* — the baseline is composition of existing primitives; a new primitive claim must clear a higher bar than that.
- **Counterfactual.** If this project had never been proposed, would the 2026 IETF agent-identity drafts + Teleport's proprietary answer fill the gap? Evidence says **yes** — they are already working the edge, and the gap is actively contested, not orphaned. *"What if this is confounded?"* — the candidate B's "primitive-ness" is confounded with adoption. `PRODUCT_THESIS.md` §P5 §6: *"It becomes a real primitive if... adopted by parties other than its creator."* That is a hypothesis contingent on a future event, not a current fact. Today's realized gravity is **A (none)**, per `ECOSYSTEM_THESIS.md` §P5 §10.
- **Placebo discipline.** Is the candidate simply a better-configured SPIRE + a delegation extension? `TECHNICAL_VALIDATION.md` §P5 §3 "Probably false: that this can be solved by 'more SPIRE configuration' rather than a new primitive." So the placebo is *probably* insufficient — but "probably" is not "definitely," and the falsification experiment (Level 2) has not run. The Empiricist must hold: novelty-unproven-by-experiment; the frozen `LEVEL0_1_FEASIBILITY_GATE.md` (~80% solved by composition) shrinks the "new" engineering surface to P-C, which the same document says may be logically impossible.
- **Survivorship-bias audit.** The Docker / Git / SPIFFE / OAuth analogies in `PRODUCT_THESIS.md` and `ECOSYSTEM_THESIS.md` are survivorship-biased. For every Docker there were many "portable X format" efforts that did *not* get adopted. The base rate of "a single small team standardizes a portable binding and gets it adopted by others against well-funded incumbents" is moderate-to-low. The Empiricist flags: the analogies are *plausible*, not *probabilistic*, and the team has been treating them as a probabilistic anchor (Cartographer's R1 lift of P5 was based on this analogy, not on measured evidence).
- **Change conditions for confidence.** Confidence: novelty-is-real **Medium** (analogical precedent — Docker, SPIFFE; component primitives all exist; standard venue is live). Shift up: a working P-C composition under the S1–S5 strengthened reading **plus** a named second-party adopter. Shift down: Level 2 falsifies the interoperability clause (S7) **or** P-C is confirmed logically impossible under S4, **or** the IETF venue names "agent identity" without incorporating the delegation layer.

**Empiricist headline finding:** the load-bearing novel claim is *not* a new item; it is a composition whose primitive-status is contingent on a future adoption event for which there is zero current evidence. The one piece of the claim that is genuinely engineering-unsolved (P-C) is, per the team's own frozen evidence, **possibly a logical impossibility** — which is the strongest form of falsification: a falsification *by the team's own prior work*.

### 4.2 Cartographer (Seat of Reasoning) — opens after Empiricist

**Mental models used:** stated vs actual claim; sunk-cost drift; naming as commitment; frame selection as hidden decision; re-asking the question; drift detection.

- **Stated vs actual claim.** The *stated* claim (`PRODUCT_DEFINITION.md`, the directive) is: "a system that lets a workload or agent prove... with bounded scope/time/revocation, without a live call to a shared, always-online authority." The *actual* claim (per `PRODUCT_THESIS.md` §P5 §3 and `LEVEL0_1_FEASIBILITY_GATE.md`) is narrower: "a *standardized binding* of three existing primitives (workload identity, attenuable capability, revocation), portable across trust domains, with primitive-status contingent on third-party adoption." These are different commitments. The stated claim sounds like a primitive; the actual claim is a *standardization* claim with an adoption contingency. The frame has drifted under evidence — the drift is *correct* (it is the team's own feasibility gate that forced the narrower claim) — but the artifact (the unfrozen `RFC-002-conceptual-domain-model.md`) was produced in *architecture mode* and its capitalized Concept objects ("Reconstruction Record," "Delegation," "Revocation," "Boundary," "Verification," "Reviewability") can be read as a re-inflation of the broader frame, even though `RFC-002` is explicitly *implementation-independent* and forbids classes/DB/schema/APIs. **Naming as commitment** — the Cartographer's named risk — is exactly the concern here. Capitalizing these as Concepts in an RFC commits the project to them as primitives before the evidence supports it.
- **Sunk-cost drift.** The R1 Cartographer (in the project's own R1 cycle, frozen in `agents/agents/journal/2026-06-19-r1-problem-ranking-application.md`) **lifted P5** (A5 1→2, SPIFFE standardization). That lift was the *only* council upgrade in R1. It was analogy-based ("Docker / SPIFFE / OCI precedent"), not measured. Subsequent investment — the falsification experiment, the feasibility gate, RFC-001, RFC-002 — has been anchored to that lift. The frozen `LEVEL0_1_FEASIBILITY_GATE.md` (~80% solved, P-C maybe impossible) is the team's *own* evidence eroding the lift, not the founder's intuition. Has the project treated that erosion honestly? The current artifact (RFC-002) is unfrozen and *implementation-independent*, but the conceptual model it names is *richer* than what the feasibility gate's evidence supports. That is the drift signal: architecture mode is *structurally biased* toward finding "concepts" where the evidence supports "compositions," and the conceptual model risks entrenching the inflated frame before the falsification question this document exists to answer is resolved.
- **Re-asking the question.** "What is the smallest novel primitive this project is attempting to create?" Re-asked honestly, after the feasibility gate: the smallest novel thing is **the composition + standardization**, classified as **compositional** novelty, contingent on adoption. There is no *item*-novelty. The question "primitive or combo?" is — at the *item* level — settled (combo), and at the *composition* level — *conditional on adoption* — the honest answer is "we are *attempting* a Docker-class compositional primitive, and the evidence for whether the compose-becomes-primitive transition is real is exactly the evidence that does not yet exist." The Cartographer does not bless "we're discovering a primitive" and does not condemn "we're merely combining" — it names the commitment honestly and flags the naming-as-commitment risk in `RFC-002` as the discipline question this project has not yet forced.

**Cartographer headline finding:** the actual claim, in one honest sentence, is "we are attempting a Docker-class compositional primitive, with primitive-status contingent on third-party adoption that has produced zero current signal." The framing of `RFC-002`'s conceptual model risks re-inflating that honest sentence back into "we are discovering a primitive" by naming as Concepts what the evidence supports only as components of a composition. The Cartographer cannot bless the conceptual model as currently framed without that naming-as-commitment risk being explicitly carried forward.

### 4.3 Red Team (Seat of Catastrophe) — opens after Cartographer

**Mental models used:** mechanism not vibe; 90-day worst plausible; reversibility; closed-loop stability; attacker-controllable inputs; failure-of-failure-detector; suggestions indistinguishable from decisions.

- **Mechanism not vibe — the mechanism of "primitive failure."** The mechanism by which the primitive claim fails is **not** a single engineering falsification event. It is a *trajectory*:
  1. The Level 2 experiment runs and (a) either confirms the feasibility gate's ~80%-solved-by-composition finding and collapses P-C to a thin revocation layer, or (b) confirms P-C is a logical impossibility (S4) and falsifies H₁ by construction. Either outcome **shrinks the primitive claim** to "a thin layer over OAuth Status List" or to "a model of an impossibility."
  2. Even if (1) is benign, the IETF venue (four 2026 drafts, named in `TECHNICAL_VALIDATION.md` §P5 §6) may converge on a draft that does *not* incorporate the project's specific shape. ECOSYSTEM_THESIS §P5 explicitly names the Teleport fragmentation risk. The primitive claim then dies not by falsification but by **indifference** — the venue forms the noun without the project.
  3. Even if (1) and (2) are both benign, adoption is contingent on third parties, and the demand side has produced **zero** signal. The two-sided market (originator-supply / adopter-demand) can produce a supply-side primitive with no demand-side adoption, which is *not a primitive* in any meaningful sense (a primitive requires `ECOSYSTEM_THESIS.md` §P5 §1–§9 to be met *through* realized adoption).
- **90-day worst plausible, not worst imaginable.** Within 90 days, the most plausible catastrophic outcome is not "the primitive is impossible" (the framework is well-tuned to detect that) but: "the experiment passes, P-C collapses to a thin layer, the conceptual model in RFC-002 entrenches, and the project discovers that the demand side has not moved — and the framework's detectors are blind to that, because they are engineering-falsification-tuned, not market-indifference-tuned." The primitive claim dies not by falsification but by indifference.
- **Reversibility.** The conceptual model in `RFC-002-conceptual-domain-model.md` is **unfrozen** (per RFC-000 Architecture Review Process state machine, per the project's own state doctrine). Structurally reversible. **But** — naming-as-commitment cost is real. Capitalized Concept objects ("Reconstruction Record," "Delegation," "Revocation") create vocabulary dependencies; walking them back is a journal entry plus a credibility cost, not a file delete.
- **Closed-loop stability.** The RFC-002 → RFC-003 → implementation loop has a closed-loop risk if each RFC entrenches the prior's assumptions without re-questioning. The **directive** (which pauses the RFC sequence after RFC-002) is exactly a destabilizer of that loop. **Positive governance signal** — the framework is operating as designed.
- **Attacker-controllable inputs.** In the primitive-claim sense, the "attacker" is the *adoption market*. Whether third parties adopt is an attacker-controllable (in the broad sense) input the project does not control. The primitive-claim's success criterion rests on a variable the project can influence but not determine. Red Team flag: **any primitive-claim that rests on "third parties will adopt" rests on an attacker-controllable input the project does not control.**
- **Failure-of-failure-detector.** The project's detection machinery (the feasibility gate, the falsification experiment, the council review) is well-tuned to engineering falsification. It is **not** tuned to adoption-indifference. If the project produces a working delegation primitive and zero third parties adopt it within 180 days, *no instrument in the framework fires*. RESEARCH_PROGRAM Wave 5 / Task 8 (unit-economics) is the closest, but it is about *pricing*, not about *adoption-signal*, and it has not run. The Red Team flags this asymmetry as the **single most important gap** in the project's detection machinery: the framework can detect "it's impossible" but cannot detect "nobody wants it," and "nobody wants it" is, per the trajectory above, the more plausible failure mode.
- **"We'll monitor it" without mechanism.** The standard deferral is "adoption will tell us." The project has no mechanism that fires on "N days passed, zero adopters." This is the Red Team's exact named red flag ("'we have observability' with no model of what's missed," `red-team.md`).
- **Silent-failure pattern (the founder-named risk).** `FOUNDER_PROBLEM_FIT.md` §P5 §5, §P5 §7: *"a delegation system that's mostly correct is arguably worse than no system, because it creates false confidence."* This is the named personal risk. The Red Team confirms: if P-B is "adopted" but the P-C limit is silent (within-window revocation during partition is not observable, the verifier doesn't know), the false-confidence failure is *load-bearing*, not edge-case. The very property the primitive is sold on (independent revocation) is the property that, under partition, is impossible. The Red Team's **silent-failure pattern is the project's own named risk**, restated in mechanism terms.

**Red Team headline finding:** the most plausible 90-day catastrophic outcome is not "the primitive is impossible" (the framework can detect that) but "the primitive is built, the framework is happy with it, and adoption never arrives" — and the framework has no instrument to detect that. **The framework's detection asymmetry is the Red Team's most important finding.** The silent-failure pattern is the project's own named risk, mechanism-resolved to "the primitive's headline property is impossible during the worst case (partition), and the operator cannot know."

### 4.4 Economist (Seat of Incentives) — opens after Red Team

**Mental models used:** who-pays-when-wrong by role; where savings go (reabsorbed vs captured); opportunity cost; attention as finite budget; revealed preference over stated intent; two-sided markets / silent loser side; defeater-test (named actor with authority to act on the insight).

- **Who-pays-when-wrong, by role.** If the compositional primitive is built and not adopted, **the founder pays** — in months. Not "the team," not "the project." The named actor is the founder, and the currency is founder-months. If the primitive is built and adopted but the silent-failure pattern (Red Team) bites, **the relying party pays** — in security incidents, breach exposure, and trust. Not the founder, not the principal, not the delegator. The *who-pays* is distributed across roles, and the founder is not on the loss-side for the security failure (the RP is) but is on the loss-side for the *non-adoption* outcome. **Different failure modes hit different actors.**
- **Where savings go, if it works.** If P-B enables least-privilege cross-org agent auth, the savings (reduced blast radius, no federation overhead) accrue to **the adopting org's security team** (lower incident cost) and **the agent builders** (easier cross-org workflows). In security tooling, savings are typically *reabsorbed* into the org (lower risk → lower headcount need → budget reallocated) — not captured by a vendor, **unless** the vendor charges per-verification or per-delegation. For an *open standard*, there is no obvious capture mechanism. The Economist names this as the structural-revenue-model risk: `RESEARCH_PROGRAM.md` Wave 5 / Task 8 (unit-economics reality check) is the project's own pre-registered acknowledgement of this risk on P5, and it has not run.
- **Originator-capture inversion.** The very property that makes P-B a *primitive* (openness, per `FOUNDER_PROBLEM_FIT.md` §P5 §10: *"a closed delegation protocol undermines its own value proposition"*) is the property that **precludes originator capture**. The economist's two-sided-market logic bites: the originator pays the cost (open-source labor, standards-body work, no IP protection), the ecosystem captures the value. This is a *virtue dressed as a flag* — the project's own stated rationale for openness is exactly the property that makes the founder's months a non-capturable cost.
- **Revealed preference.** The founder's sequential investment reveals a preference for the *open-standard / primitive* path over the *proprietary / product* path (Teleport-style). Revealed preference is informative. But it is a preference, not evidence that the primitive will be adopted.
- **Opportunity cost.** The months spent on RFC-002 → RFC-003 → implementation are months not spent on a capture-allowing framing (even on P5, a Teleport-style proprietary partial answer would allow capture). The Economist surfaces: even *within* P5, there is a framing the team has *explicitly* excluded (the closed-source / proprietary / hosted framing) which is the only framing that allows the founder to capture return on the labour.
- **Two-sided market / silent loser side.** In an adoption market, the **adopter side is the silent side** — the founder's labour cannot make adopters appear. The Economist flags: this is a two-sided market (originator-supply / adopter-demand) and the demand side has produced **zero** signal — no adopter, no letter of intent, no pilot. A two-sided market with zero demand-side signal is, by the Economist's doctrine, the highest-risk configuration.
- **Defeater-test (named actor with authority to act on the insight).** The insight — "you are paying a certain cost in founder-months to produce a non-capturable surplus whose demand side is silent" — needs an actor who can act. The founder is that actor. The Economist does **not** say "don't do it" — it says "recognize this is a *contribution-shaped commitment with product-shaped cost*, and the only honest justification is reputational/portfolio or genuine belief that the standardization venue will adopt — neither of which is a financial return."

**Economist headline finding:** the *value-capture structure is inverted relative to a product.* The founder bears the certain cost; the ecosystem would capture the surplus if adoption occurred; the demand side has produced zero signal. The Economist's defeater is met (the founder is the named actor-of-decision). The cost is certain; the return is contingent and non-capturable. The Founder named this asymmetry implicitly by choosing the open path, and the analysis names it explicitly: the project is, from the Economist's view, a *contribution* that is being paid for as if it were a *product*.

### 4.5 Operator (Seat of Work) — opens after Economist

**Mental models used:** actual day of actual user; adaptive vs technical work; ironies of automation; the two users (senior vs median); invisible expertise; adoption as sociotechnical system (tool is 30%); adoption kill; defeater-test (actual adoption data from comparable prior tool with same substrate).

- **Actual day of actual user.** A platform engineer at a company running SPIFFE needs to let an AI agent in domain A act on behalf of a principal in domain B, cross-org, without federation. Today they (a) set up SPIFFE Federation (expensive, per `TECHNICAL_VALIDATION.md` §P5 §3 "everyone wants SPIFFE, almost no one can afford to build it right"), (b) use a long-lived API key (insecure), or (c) use Teleport's proprietary thing (lock-in). The day-of is real, concrete, and underserved. P-B addresses a *real* Tuesday, not a hypothetical one.
- **Adaptive vs technical work.** Adopting a new identity primitive is **not** a tool-install. It is a *sociotechnical migration*: the user has to change how their org reasons about identity (from static creds to attenuable delegations). Adaptive work, not technical work. The tool is 30%; the adaptive shift is 70%. The project's primitive-framing underweights this.
- **The two users — senior vs median.**
  - The **senior** security engineer can integrate P-B correctly, understands the S4 limit, and designs around it (revocation-as-eventual, post-partition-recovery).
  - The **median** platform engineer — the one the ecosystem says "almost no one can run SPIFFE correctly" — reads the conceptual model, sees "Revocation: independently revocable," and trusts it in the partition case, getting burned. It makes the *hard* part (debugging a multi-hop chain at 3am during a partition; reasoning about whether a revocation propagated) **harder at the worst moment**, because the cryptographic, offline, multi-hop chain is opaque to the operator during incident response. The easy 90% is made easier; the hard 10% is made harder at the worst moments. **Classic irony of automation.**
- **Invisible expertise.** The expertise required to *safely* operate P-B correctly — knowing the partition case is impossible, designing the system so revocation is treated as eventual post-recovery — is *invisible* in the conceptual model. The model says "Revocation" as a clean primitive; the invisible expertise is "you must not rely on it during partitions." The conceptual model hides the very expertise that makes deployment safe.
- **Adoption kill — prerequisite-stacking.** The Operator identifies the **specific adoption kill**: P-B sits *on top of* SPIFFE in difficulty. The population that can adopt P-B is a strict subset of the population that has correctly adopted SPIFFE — which the ecosystem itself describes as "almost no one." P-B adds adoption cost (an additional layer of identity reasoning), and the population that clears all the prerequisites (correctly-run SPIFFE + delegation reasoning + partition-revocation reasoning + two-org coordination) is **vanishingly small today**. The adoption kill is *prerequisite-stacking on an already-underserved population*. The Red Team's silent-failure pattern compounds this: the median user, adopting the prerequisite, already does not understand S4; adding P-B on top does not fix that, it *exploits* it (the conceptual model's "Revocation" Concept creates false confidence for the median user).
- **Defeater-test (comparable prior adoption data).** The Operator's named defeater is "actual adoption data from a comparable prior tool with the same substrate." The comparable prior tools:
  - **Macaroons** (2014, Google): 12 years old; niche adoption; not an industry standard.
  - **Biscuit**: actively maintained; niche format; not an industry standard.
  - **SPIFFE/SPIRE**: graduated CNCF; described as "everyone wants SPIFFE, almost no one can afford to build it right" (`TECHNICAL_VALIDATION.md` §P5 §3 verbatim); slow adoption; high operational cost.
  - **SPIFFE Federation**: the closest comparable (cross-trust-domain, same substrate); operationally expensive and incomplete per the same source.
  - The Operator's defeater is met with **negative evidence**: the comparable prior tools in this substrate family (capability tokens, SPIFFE, Federation) have *slow, partial, expert-gated* adoption, **not viral adoption**. Projecting that pattern onto P-B: adoption will be slow, expert-gated, and bottlenecked on the population that can operate its prerequisite (SPIFFE) correctly — which is itself the population the ecosystem describes as struggling.

**Operator headline finding:** the reachable-adopter population is the intersection of (can-run-SPIFFE-correctly) ∩ (wants-delegation), and the first set is already described as underserved by the current primitive it sits on. Prerequisite-stacking on an already-underserved population is the named adoption kill. Comparable prior tools in this substrate family show slow, expert-gated, not-viral adoption. The conceptual model's capitalized "Revocation" Concept risks creating false confidence for the median user (the silent-failure pattern the Red Team and `FOUNDER_PROBLEM_FIT.md` §P5 §5 name). The Operator's defeater is met with **negative comparable-adoption evidence**.

## 5. Category Test — "This project is to ___ what Docker was to containers"

The Docker precedent is: a combination of existing primitives (chroot, cgroups, union filesystems) named and standardized into a portable, addressable image format, that the ecosystem adopted as a substrate. The novelty was the *frame / naming / adoption*, not the technology.

**Generating candidate completions of the sentence.** The directive requires at least five candidate statements. Three are rejected as weak; four are kept, with the strongest single one highlighted.

### 5.1 Rejected (weak) candidates

- **C1: "…to **workload delegation** what Docker was to containers."** *Rejected.* "Workload delegation" is already a nameable concept (RFC 8693, SPIFFE Federation, AWS IAM Roles Anywhere all use it). Docker named an *unnamed* frame; "workload delegation" is already named. The candidate does not pass the "name something not currently a single noun" test.
- **C3: "…to **offline verifiable delegation** what Docker was to containers."** *Rejected.* "Offline verifiable delegation" within a domain is *already* what Biscuit is. The cross-trust-domain + no-shared-broker part is the actually-novel part; this candidate is too broad and under-claims.
- **C7: "…to **identity** what Docker was to containers."** *Rejected.* Too broad. Identity is SPIFFE's noun (`ECOSYSTEM_THESIS.md` §P5 §2: SPIFFE is the existing primitive at this slot). The candidate fails to name a *new* slot.
- **C8: "…to **authorization** what Docker was to containers."** *Rejected.* Too broad. Authorization is Zanzibar's / OPA's noun. The candidate fails to name a *new* slot.

(Three of these rejections suffice to demonstrate weak-candidates-rejected discipline; the directive asks for at least five candidates *total*, which is met by the four kept + four rejected = eight below.)

### 5.2 Kept (evidence-supported) candidates

- **C2: "…to **attenuable cross-domain capability** what Docker was to containers."** *Kept provisionally.* "Attenuable cross-domain capability" is a compound that is not currently a single noun. But the "attenuable capability" half is macaroon/biscuit's existing noun, and the project would be naming the *cross-domain* half of the compound. Stronger than C1/C3 but somewhat undersells the "no shared authority" clause, which is what distinguishes P-B from the insufficient-adjacent cluster (Vault, AWS IAM Roles Anywhere, SPIFFE Federation).
