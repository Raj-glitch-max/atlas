## 2. Repository Design Principles

1. **Entry points are few and obvious.** An engineer opening the repository should need to read fewer than five files to be fully oriented. Root files are entry points, not documentation storage.

2. **Frozen content is immutable by process, not by convention.** `make check-frozen` + `FROZEN.sha256` + `scripts/frozen-docs.list` form a mechanical gate. A frozen document is only changed through a journal entry, a dated change-note, a baseline regeneration, and a co-committed hash update.

3. **AI memory and human documentation are separate.** `context/` is loaded by AI sessions to understand current project state. `docs/` is human engineering documentation. The two serve different audiences and must not conflate.

4. **The laboratory is hermetically isolated.** `lab/` runs pre-registered experiments. It does not contain product code, architecture decisions, or product-management work. Nothing enters the lab without a frozen pre-registration.

5. **The reasoning framework is self-contained.** `agents/` houses all council, domain, working specialist, journal, and governance files. External sessions import from `agents/` via the `CLAUDE.md` reading order. The framework does not depend on any other directory for its operation.

6. **History is preserved, not rewritten.** `archive/` and `docs/project/PROJECT_HISTORY.md` intentionally preserve old names, old absolute paths, and superseded structures. These files are not updated to reflect current structure.

7. **The `rfc/` directory is for architecture-level decisions.** RFCs are not created without explicit founder authorization. Existing drafts are in progress; they are not frozen.

8. **Two root files are permanent anchors.** `LEVEL0_1_FEASIBILITY_GATE.md` and `P5_FALSIFICATION_EXPERIMENT.md` live at root permanently. They are referenced by relative path from inside frozen lab documents that cannot be amended without triggering a full governance cycle. The engineering cost of moving them exceeds the organizational benefit.

## 3. Entry-Point Reading Order

For a new AI session, load in this order:

1. `context/08_AI_HANDOFF.md` — current pipeline state and next actions. **Always read first.**
2. `context/00_PROJECT_CONTEXT.md` — project mission, what is and is not being built.
3. `CLAUDE.md` — constraints, working conventions, and forbidden behaviors.
4. `agents/GOVERNANCE.md` — binding rules for agent interaction.
5. `agents/journal/` — committed decisions. Start with the two existing entries.

For a human engineer, read in this order:

1. `README.md` — orientation and quickstart.
2. `CONTRIBUTING.md` — how to commit, what is frozen, how to amend.
3. `PROJECT_STRUCTURE.md` — canonical directory map and ownership.
4. `DEVELOPMENT_RULES.md` — branch, commit, freeze, journal, and RFC policy.

## 5. Repository Invariants

These conditions must hold at every commit. `make ci` enforces them mechanically where noted.

1. **Frozen-doc integrity passes.** `make check-frozen` exits 0. Any frozen file changed without a corresponding journal entry, dated change-note, and baseline regeneration is a violation. *(Enforced by CI.)*

2. **No secrets in tracked files.** `make secrets` (gitleaks) exits 0. A committed secret requires secret rotation and history rewrite. *(Enforced by CI.)*

3. **Markdown lint passes.** `make docs-lint` exits 0. *(Enforced by CI.)*

4. **Pre-commit hooks pass.** `make lint` exits 0. *(Enforced by CI.)*

5. **Commit messages conform to Conventional Commits.** The `commit-msg` hook enforces the format. *(Enforced by hook.)*

6. **`lab/` contains no product code.** The laboratory runs pre-registered experiments and records evidence. Product code, architecture design, and product-management artifacts do not belong in `lab/`.

7. **`context/` is not used as general documentation storage.** `context/` is AI session memory. New documentation goes in `docs/`, not `context/`.

8. **`agents/` is self-contained.** No file outside `agents/` depends on a specific file inside `agents/` for its mechanical operation (only for reference). Reorganizing `agents/` internals does not break the repository.

9. **`LEVEL0_1_FEASIBILITY_GATE.md` and `P5_FALSIFICATION_EXPERIMENT.md` remain at root.** They are permanent anchors. Relocation requires a lab-document freeze amendment cycle for every frozen lab file that references them.

10. **`docs/` is organized by audience and lifecycle.** `docs/product/`, `docs/engineering/`, `docs/research/` contain frozen content. `docs/project/` contains living project documentation.

## 6. Future Repository Rules

**When stack is selected:**
- Add a language-specific package manifest, linter, formatter, and test runner.
- Add stack-specific CI job for build and test artifacts.
- Extend `CONTRIBUTING.md` §5 with the stack details.
- Replace `tests/README.md` with a real test suite.
- Add a product `Dockerfile` (the current `Dockerfile` is the dev toolbox, not a product image).

**When new frozen documents are created:**
- Add their paths to `scripts/frozen-docs.list`.
- Run `make frozen-baseline` immediately after.
- Commit `FROZEN.sha256` in the same commit as the new document.

**When RFCs advance:**
- A RFC transitions from Draft to Accepted via a founder decision recorded in `agents/journal/`.
- An Accepted RFC may be frozen if engineering consensus requires it.
- New RFCs require explicit founder authorization per `DEVELOPMENT_RULES.md` RFC Policy.

**When `agents/` evolves:**
- Working specialists are promoted, demoted, or retired per `agents/GOVERNANCE.md` §8.
- Journal entries record every meaningful decision, including promotions.
- New domain anchors require founder opt-in via journal entry.

**When `lab/` runs an experiment:**
- Pre-registration is required before any experiment begins (see `lab/EXPERIMENT_CHECKLIST.md`).
- Evidence is filed under `lab/evidence/<EXP-NNN>/`.
- `lab/EXPERIMENT_LOG.md` is append-only. Never delete or edit past entries.
- Lab decisions are surfaced to the founder before feeding back into product or engineering.

## 7. Repository Organization Is Frozen

Repository organization is now considered **frozen** until engineering explicitly requires changes.

The following are the only legitimate reasons to change the repository structure after this commit:

1. **A technology stack is selected.** This will require adding new top-level directories (e.g., `src/`, `cmd/`, `pkg/`) and extending CI.
2. **A lab experiment produces an artifact** that requires a new evidence directory or data structure under `lab/evidence/`.
3. **An RFC is accepted and promoted** to a specification that requires a new location.
4. **A frozen document requires a governance-approved amendment** that changes its path.
5. **A security or engineering finding** reveals that the current structure creates a material risk.

Any other change to directory structure or root-level file placement is outside scope and requires an explicit founder decision recorded in `agents/journal/`.

**Documentation mode is closed. Engineering is the default mode from this point forward.**
